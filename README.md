# 23种设计模式
## 一、设计模式的定义
设计模式是一套被反复使用、多数被人知晓的、经过分类编目的、代码设计经验的总结，使用设计模式是为了重用代码、让代码更容易被他人理解、提高代码的可靠性。

## 二、设计模式的优点
- 提高软件系统的开发和软件质量，在一定程度上节约设计成本。
- 使得设计方案更加灵活，且易于修改。

## 三、设计模式的两大主题
**系统复用与系统扩展**

## 四、分类
### 创建型模式(5)
#### 主要用于创建对象
- **简单工厂模式**：选择实现
  - **作用**：将类实例化的操作与使用对象的操作分开，让使用者不用知道具体参数就可以实例化出所需要的产品类。
  - 何时使用?
    - 完全封装隔离具体实现，让外部只能通过接口来操作封装体。
    - 把对外创建对象的职责集中管理和控制。
  - 优点：
    - 将客户端和具体实现解耦，客户端无需知道所创建具体产品类的类名，只需要知道具体产品类所对应的参数。
    - 工厂类含有必要的判断逻辑，以决定什么时候创建哪个产品类，客户端可以免除直接创建产品对象的责任。**实现对责任的分割**，提供了专门的工厂类用于创建对象。
  - 缺点：
    - 工厂类权力过于集中。
    - 不符合开放封闭原则，增加新的产品需要修改工厂类的逻辑。
    - 不够灵活，难扩展。 
- **工厂方法模式**：延迟到子类来选择实现
  - **作用**：定义一个用于创建产品的接口，由子类决定生产什么产品。 
  - 何时使用?
    - 如果一个类需要创建一个接口的对象，但是又不知道具体的实现，使用工厂方法模式把创建对象工作延迟到子类去实现。
    - 如果一个类本身就希望由它的子类来创建所需对象。 
  - 优点：
    - 向客户端隐藏具体产品类被实例化这一细节，创建一个新产品时，只需要提供创建产品的工厂，无需关注创造细节，甚至不需要了解产品类的类名。
    - 创建一个新产品时，只需要增加具体工厂类和产品类，而不需修改原有代码，符合开放封闭原则。
  - 缺点：
    - 创建一个新产品时，需要增加具体工厂类和产品类，系统中类的数量会成对增加，会增加系统的复杂性。
  - 工厂方法模式与简单工厂模式
    - 工厂方法模式实现时，客户端需要决定是实例化哪个工厂，有选择判断的问题，工厂方法把简单工厂的内部逻辑判断转移到了客户端代码来进行。 
    - 工厂方法类的核心是一个抽象工厂类，而简单工厂模式把核心放在一个具体类上。
- **抽象工厂模式**：选择产品族的实现
  - **作用**：提供一个创建产品族的接口，其每个子类可以生产一系列相关的产品。
  - 何时使用?
    - 希望一个系统独立于它的产品的创建。
    - 系统中有多于一个的产品族。
    - 系统提供一个产品类的库，所有的产品以同样的接口出现，从而使客户端不依赖于具体实现 
  - 优点：
    - 符合开放封闭原则。增加新的具体工厂类无需修改已有代码。
    - 符合单一职责原则。将生成产品的代码抽取到同一位置，使得代码易于维护。
    - 可以避免客户端和具体产品代码的耦合。
    - 切换产品族变得容易。
    - (以一种倾斜的支持增加新的产品) 
  - 缺点： 
    - 引入众多的类和接口，代码会很复杂。
    - 不太容易扩展新的产品。
  - 抽象工厂模式与工厂方法模式
    - 工厂方法模式一般是针对单独的产品对象的创建，而抽象工厂模式注重产品族对象的创建，这是他们的区别。 如果把抽象工厂创建的产品族简化，这个产品族只有一个产品，那么抽象工厂与工厂方法是差不多的。抽象工厂可退化为工厂方法，工厂方法可退化为简单工厂。 
- **建造者模式**：分离整体算法构建和部分构造
  - **作用**：将一个复杂对象分解成多个相对简单的部分，然后根据不同需要分别创建它们，最后构建成该复杂对象。
  - 何时使用?
    - 相同的构建过程有不同的表示。
    - 创建对象的算法，应该独立于该对象的组成部分以及它的装配方式。  
  - 优点
    - 客户端不必知道产品内部的组成细节，将**产品本身和产品的创建过程解耦**，使得**相同的创建过程可以创建出不同的对象**。
    - 可以更加精细地控制产品的创建过程。将复杂产品的创建步骤分解在不同的方法中，使得创建过程更加清晰，也更加方便使用程序来控制创建过程。
    - 增加新的具体建造者无需修改原有类库代码，符合开放封闭原则。
  - 缺点
    - 建造者模式所创建的产品一般都具有较多共同点，其组成部分相似。如果产品的差异性较大，则不适合建造者模式，因此其使用范围限制。
    - 如果产品内部变化复杂，可能会导致需要定义很多具体建造者来实现这种变化，导致系统变得庞大臃肿。
  - 建造者模式和抽象工厂模式
    - 抽象工厂模式的目的是创建产品族，这个产品族里面的单个产品，就相当于是构成一个复杂对象的部件对象，抽象工厂对象创建完成后立即返回整个产品族；而建造者模式的目的是按照算法构建，一步一步来构建一个复杂对象，通常要整个构建过程结束后才会得到最终的产品对象。
- **原型模式**：克隆生成对象
  - **作用**：将一个对象作为原型，通过对其进行复制而克隆出多个和原型类似的新实例。
  - 何时使用?
    - 一个系统想要独立于他想要使用的对象时。
    - 希望动态的增加或减少产品类。
  - 优点：
    - 简化对象的创建过程，提高效率。
    - 对客户隐藏具体的实现类型。
    - 在运行时动态改变具体的实现类型。
  - 缺点：
    - 每个类都需要配备克隆方法，对已有类改造不符合开放封闭原则。
    - 带有引用对象的类进行克隆时需要深克隆，实现困难。
- **单例模式**：控制实例数目
  - **作用**：某个类只能生成一个实例，该类提供了一个全局访问点供外部获取该实例。
  - 何时使用?
    - 系统只需要一个实例对象。
    - 客户端调用类的单个实例只允许使用一个公共访问点，除了该公共访问点，不能通过其他途径访问该实例。
  - 优点：
    - 每个类只创建一个对象，节约资源
    - 可以控制创建实例的数目。
  - 缺点：
    - 单例类的职责过多，不符合单一职责原则。
  - 扩展：懒汉式和饿汉式
### 结构型模式(7)
#### 主要用于处理类和对象的组合
- **适配器模式**：转换匹配，复用功能
  - **作用**：将一个类的接口转换成客户希望的另外一个接口，使得原本由于接口不兼容而不能一起工作的那些类能一起工作。
  - 何时使用?
    - 系统需要使用现有的类，而这些类的接口不符合系统的需要。 
  - 优点：
    -  将目标类和适配者类解耦，通过引入适配器类重用现有适配者类，而无需修改原有代码。
    -  增加了类的透明性和复用性，将具体的实现封装在适配者中，对于客户端是透明的，且提高了适配者的复用性。
    -  更好的灵活性和扩展性。
  - 缺点：
    - 代码整体复杂度增加。 
- **桥接模式**：分离抽象和实现
  - **作用**：将抽象与实现分离，使它们可以独立变化。它是用组合聚合关系代替继承关系来实现，从而降低了抽象和实现这两个可变维度的耦合度。
  - 何时使用?
    - 对于那些不希望使用继承或因为多层次继承导致系统类的个数急剧增加的系统，桥接模式尤为适用。 
    - 一个类存在两个独立变化的维度，且这两个维度都需要进行扩展。
  - 优点：
    - 分离抽象和实现部分。
    - 桥接模式有类似多继承的方案。但是多继承不符合单一职责且类太多，桥接模式是一种比多继承更好的方案，符合合成复用原则。
    - 具有良好的可扩展性，符合开放封闭原则。在两个维度中任何一个维度增加产品都不需要修改原有代码。
    - 实现对用户透明，对用户隐藏细节，用户不需要关心实现，在抽象层通过聚合关联关系完成封装和对象的组合。
  - 缺点：
    - 桥接模式的引入增加了系统的理解和设计维度。
- **组合模式**：统一叶子对象和组合对象。
  - **作用**：将对象组合成树状层次结构，使用户对单个对象和组合对象具有一致的访问性。
  - 何时使用?
    - 系统为了体现部分与整体层次的结构时，希望用户可以忽略组合对象和叶子对象的不同，统一地使用组合结构中的所有对象。 
  - 优点：
    - 增加新的元素无需修改原有代码，符合开放封闭原则。
    - 统一了叶子对象和组合对象。客户端调用简单，无需关注自己创建的是叶子对象还是组合对象。 
  - 缺点：
    - 在添加时，很难限制组件的类型。需要检测组件类型的时候，使得我们不能依靠编译期的类型约束来完成，必须在运行期间动态检测。 
  - 扩展：安全组合模式和透明组合模式。 
- **装饰模式**：动态组合，动态是手段，组合是目的
  - **作用**：动态的给对象增加一些职责，即增加其额外的功能。
  - 何时使用?
    - 如果不需要影响其他对象的情况下， 以动态、透明的方式给对象添加职责、
    - 如果不合适使用子类来进行扩展时(继承)，可以考虑装饰模式。 
  - 优点：
    - 装饰模式与继承关系都是扩展对象的功能，但是装饰模式可以提供比继承关系更加的灵活。
    - 通过使用不同装饰类的排列组合，可以创造出不同行为的组合。 
    - 更容易复用功能，
    - 简化高层定义。
  - 缺点： 
    - 会产生很多细粒度对象。 
  - 装饰模式和策略模式：
    - 装饰模式改变的是对象的外表，而策略模式改变的是对象的本质。装饰模式是递归调用，策略模式是一次调用。  
- **外观模式**：封装交互，简化调用
  - **作用**：为多个复杂的子系统提供一个一致的接口，使这些子系统更加容易被访问。
  - 何时使用?
    - 在维护一个大型的遗留系统的时候。 
    - 当客户端与抽象类的实现部分之间存在极大的依赖性的时候。
  - 优点：
    - 实现了子系统和客户端的松耦合关系，当子系统发生变化时，无需通知客户端，由外观类直接调用。 
  - 缺点： 
    - 增加新的子系统需要修改外观类，不符合**开放封闭原则**。
- **享元模式**：分离与共享
  - **作用**：运用共享技术来有效地支持大量细粒度对象的复用。
  - 何时使用?
    - 如果一个应用程序有大量的对象，而大量的对象造成了很大的存储开销。
  - 优点：
    - 极大减少内存中对象的数量，使得相同对象或相似对象在内存中只保存一份。
  - 缺点：
    - 需要分离内部状态和外部状态，程序逻辑复杂化。
    - 需要将享元的状态外部化，读取时间加长。
- **代理模式**：控制对象访问
  - **作用**：为某对象提供一种代理以控制对该对象的访问。即客户端通过代理间接地访问该对象，从而限制、增强或修改该对象的一些特性。 
  - 何时使用?
    - 远程代码，虚拟代理，保护代理，智能指引代理。 
  - 优点：
    - 代理模式协调了调用者和被调用者，在一定程度上降低了系统的耦合。
  - 缺点：
    - 实现代理模式需要额外的工作，所以有些代理模式会变得很复杂。
### 行为模式(11)
#### 主要用于描述对类或对象怎样交互和怎样分配职责
- **职责链模式**：分离职责，动态组合
  - **作用**：请求从链中的一个对象传到下一个对象，直到请求被响应为止。通过这种方式去除对象之间的耦合。 
  - 何时使用?
     - 有多个对象可以处理同一请求，具体哪个对象处理该请求由运行时刻自动确定。客户端只需要将请求提交到链上，无须关心请求的处理对象是谁以及它是如何处理的。
     - 可以动态指定一组对象处理请求。客户端可以动态创建职责链来处理请求，还可以动态改变链最终处理的先后顺序。
     - 在不明确指定接收者的情况下，向多个对象中的一个提交一个请求。
  - 优点：
    - 请求者和接收者松散耦合。
    - 动态组合职责。
  - 缺点：
    - 产生很多细粒度对象。
    - 不一定能被处理。
      - 需要提供默认处理 
  - 职责链模式和状态模式
    - 状态模式是一个对象的内在状态发生改变，一个对象，相对比较稳定，处理完一个对象下一个对象的处理一般都已确定。在某个状态中已经确定了下一个状态（写死了，如果需要修改，需要打开每个类，进行修改，不符合开放封闭）
    - 职责链模式过于灵活，在客户端使用时，需要环境去确定下一个对象是谁，进行一系列的set操作……。职责链模式是多个对象之间的改变，就会出现某个对象不存在的情景。
- **命令模式**：封装请求
  - **作用**：将一个请求封装为一个对象，使发出请求的责任和执行请求的责任分割开。
  - 何时使用?
    -  系统需要将请求调用者和请求接收者解耦，使得调用者和接收者不直接交互。请求调用者无需知道接收者的存在，也无需知道接收者是谁，接收者也无需关心何时被调用。
    -  系统需要支持命令的撤销操作和恢复操作。
  - 优点：
    - 能较容易地设计一个命令队列。
    - 可以容易实现撤销undo和重做redo。
    - 允许接受请求的一方决定是否要否决请求。
    - 在需要的情况下，可以较容易地将命令记入日志。
    - 增加新的命令类不影响其他类。符合开放封闭原则。 
  - 缺点：
    - 使用命令模式可能会导致某些系统有过多的具体命令类。因为针对每一个命令都需要设计一个具体命令类，所以某些系统可能需要大量具体命令类，这将影响命令模式的使用。
  - 命令模式和备忘录模式：
    - 可以结合使用，实现undo功能。 
- **中介者模式**：封装交互
  - **作用**：定义一个中介对象来简化原有对象之间的交互关系，降低系统中对象间的耦合度，使原有对象之间不必相互了解。
  - 何时使用?
    - 系统中对象之间存在复杂的引用关系，产生的相互依赖关系结构混乱且难以理解。
    - 想通过一个中间类来封装多个类中的行为，而又不想生成太多的子类。 
  - 优点：
    - 它通过将对象间的通信封装到一个类中，将多对多的通信转化为一对多的通信，降低了系统的复杂性。
    - 便于集中管理。
  - 缺点：
    - 控制的集中化，于是把交互复杂性变为了中介者的复杂性。
  - 中介者模式和观察者模式
    - 可以组合使用。来实现当同事对象发生改变的时候，通知中介对象，让中介对象去进行与其他相关对象的交互。
  - 中介者模式外观模式
    - 外观模式和中介者模式有着共同的目的。它们都是调配一组相互耦合的对象的通信。不过外观模式关注的是这组复杂对象如何与外界通信，而中介者模式是调配这组复杂对象之间的通信。  
- **备忘录模式**：保存和恢复内部状态
  - **作用**：：在不破坏封装性的前提下，获取并保存一个对象的内部状态，以便以后恢复它。
  - 何时使用?
    - 适用于功能比较复杂，但需要维护或记录属性历史的类。 
  - 优点：
    - 实现状态还原。
    - 实现了信息封装。 
  - 缺点：
    - 资源消耗大。
  - 备忘录模式和原型模式
    - 可以组合使用。在原发器对象（发起人对象）创建备忘录对象的时候，如果原发器对象中全部或者大部分的状态都需要保存，一个简洁的方式就是直接克隆一个原发器对象。也就是说，这个时候备忘录对象里面存放的是一个原发器对象的实例。  
- **观察者模式**：触发联动
  - 作用：多个对象间存在一对多关系，当一个对象发生改变时，把这种改变通知给其他多个对象，从而影响其他对象的行为。
  - 何时使用?
    - 当一个对象必须通知其他对象，但是又希望这个对象和其他被通知的对象是松散耦合的。
    - 如果在更改一个对象时，需要同连带改变其他的对象，而且不知道究竟有多少个对象需要被连同改变。
    - 当一个抽象模型有两个方面，其中一个方面的操作依赖于另一个方面的变化。 
  - 优点：
    - 观察者模式实现了观察者和目标之间的抽象耦合
    - 观察者模式实现了动态联动。
    - 观察者模式支持广播通信。 
  - 缺点：
    - 可能引起无谓的操作 
- **状态模式**：根据状态来分离和选择行为。状态模式是状态驱动，由上下文负责。
  - **作用**：允许一个对象在其内部状态发生改变时改变其行为能力。
  - 何时使用?
    - 当一个对象的行为取决于他的状态，并且必须在运行时刻根据状态改变它的行为。
    - 一个操作含有庞大的条件分支语句。
  - 优点：
    - 简化应用逻辑控制。
    - 更好的分离状态和行为。
    - 更好的扩展性。
    - 显示化进行状态转换。
  - 缺点：   
    - 结构和实现复杂。
    - 不符合开放封闭原则。 
  - 状态模式与策略模式
    - 状态模式和策略模式有很大程度上的相似：它们都有一个Context(上下文/环境)类，都是通过委托（组合）给一个具有多个派生类的多态基类实现Context的算法逻辑，两者最大的差别就是状态模式中派生类持有指向Context的引用，并通过这个引用调用Context中的方法，但在策略模式中就没有这种情况。 
- **策略模式**：分离算法，选择实现。
  - **作用**：定义了一系列算法，并将每个算法封装起来，使它们可以相互替换，且算法的改变不会影响使用算法的客户。
  - 何时使用?
    -  出现同一个算法，有很多不同的实现。
    -  出现抽象一个定义了很多行为的类，并且是通过多个if-else语句来选择这些行为的情况。
    -  需要封装算法中，与算法相关的数据的情况。
    -  出现了有许多相关的类，仅仅是行为有差别的情况。
  - 优点：
    - 定义一系列算法。
    - 避免多重条件语句。
    - 更好的扩展性。 
  - 缺点：
    -  客户必须了解每种策略的不同。
    -  增加了对象的数目。
  - 策略模式与简单工厂的结合(不使用配置文件)
    - 将选择工厂的逻辑放到环境类中。
- **模板方法模式**：固定算法骨架。
  - **作用**：定义一个操作中的算法骨架，而将算法的一些步骤延迟到子类中，使得子类可以不改变该算法结构的情况下重定义该算法的某些特定步骤。
  - 何时使用?
    - 需要固定算法骨架，实现一个算法的不变的部分，并把可变的行为留给子类来实现。
    - 各个子类中具有公共行为，抽取出来，集中在一个公共类中实现，从而避免代码重复。
    - 需要控制子类扩展的情况。
  - 优点：
    - 实现代码复用。 
  - 缺点：
    - 算法骨架不容易升级。 
- **访问者模式**：预留通路，回调实现。
  - **作用**：在不改变集合元素的前提下，为一个集合中的每个元素提供多种访问方式，即每个元素有多个访问者对象访问。**将处理从数据结构中分离。**
  - 何时使用?
    - 对象结构中各个元素对应的类很少变化，但经常需要在此对象结构上定义新的操作。
  - 优点：
    -  增加新的访问操作无需修改原有代码，符合开放封闭原则。
    -  将有关的行为集中到一个访问者对象中，而不是分散到一个个的节点元素类中。
    - (双重分派)。
  - 缺点：
    - 增加新的元素很困难。每增加一个新的元素就要在抽象访问者中增加一个新的抽象操作，并在每个具体访问者类中增加相应的具体操作。
    - 破环封装。访问者模式要求访问者对象访问并调用每一个元素对象的操作，这意味着元素对象有时候暴露一些自己的内部操作和内部状态，否则无法供访问者访问。
